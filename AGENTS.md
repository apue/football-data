# Agent Operating Guide

This repository is designed for deterministic data updates with agent-assisted maintenance.

## Core Rule

Keep the pipeline reproducible. Agents may repair discovery, fetching, parsing, examples, and docs, but extracted data must be generated by committed scripts.

## Data Architecture Constraint

Treat FIFA PMSR source documents and their manifests as the canonical source for PMSR-derived technical, physical, and tactical data. FIFA public match timeline API data is an official supplemental event source for goals, assists, and goal involvements that PMSR PDFs do not expose as structured fields. SQLite databases, demo pages, notebooks with outputs, and update summaries are rebuildable artifacts.

## Legal Notice

This project uses publicly accessible FIFA Training Centre match-report pages, PMSR PDF links, and public match timeline data. Running the fetch/update workflow does not require private credentials, login access, or special authorization from this repository.

Keep the boundary clear: cite FIFA as the original source, preserve source URLs and hashes for traceability, do not bypass access controls, and do not commit or redistribute raw PDF caches unless the upstream rights holder permits it. The committed database and pages are derived analytical artifacts, not a replacement for the original source documents.

Breaking schema changes are allowed when they improve correctness or analysis quality, especially early in the project. When this happens:

- bump the schema version,
- rebuild `data/latest.sqlite` from active source documents,
- document the breaking change in `CHANGELOG.md`,
- keep source URL, source hash, parser version, and extraction timestamp available for traceability.

Prefer stable query surfaces, but do not preserve a poor schema at the cost of correctness.

Player-level tables should join primarily on `match_key`, `team`, and `player_no`. Keep `player_name` for readability and fallback matching, but do not treat names as stable primary keys across documents, languages, or report versions.

When parsing wide PMSR tables, prefer PDF text object coordinates over plain extracted line text. Plain text can omit zero-valued table cells and shift values into the wrong columns.

## Daily Pipeline

Run:

```bash
python scripts/update_dataset.py
python -m pytest
```

Expected generated outputs:

- `data/latest.sqlite`
- `manifests/latest-run.json`
- `manifests/timeline-run.json` when timeline backfill is run directly
- `manifests/sources.json`
- `manifests/discovered-sources.json`
- `manifests/update-events.json`
- `site/index.html`

## Editor's Choices

When the user asks to publish or check Editor's Choices, daily picks, 每日精选, Player of the Day, or hidden gems, use the repo-scoped skill at `.agents/skills/publish-editors-choices/SKILL.md`.

Use `scripts/prepare_editorial_packet.py --date YYYY-MM-DD` as the default local handoff entrypoint. It writes the deterministic audit packet under `agent-runs/YYYY-MM-DD/` without publishing site artifacts. Immediately run `scripts/inspect_editorial_day.py --date YYYY-MM-DD` to write `editorial_fact_pack.json` and `editorial_fact_pack.md`; use that fact pack for goal timelines, own goals, assists, team pressure, goalkeeper checks, and metric-led candidate traps instead of ad hoc SQL. Local Codex should then run bounded selection and copy review loops: write `selection_rounds/round_N/selection_decision.json` plus `selection_review.json`, revise until selection review passes or three rounds are exhausted, then write `copy_rounds/round_N/copy.json` plus `copy_review.json`, revise until copy review passes or three rounds are exhausted. Promote passing loops with `scripts/promote_editorial_loop.py --date YYYY-MM-DD`, then compile with `scripts/compile_local_editorial.py --date YYYY-MM-DD`. GitHub Actions should fetch/update data; editorial publication is manual-first because the final product needs human review.

The active production experiment is `bounded_editorial_loop_v1`, using the `bounded_selection_copy_loop_v1` workflow variant. Stable mainline responsibilities are PMSR/FIFA timeline sourcing, deterministic ranking and candidate-pool construction, deterministic editorial fact-pack construction, candidate-pool-only local selection, deterministic selection validation, bounded selection review/revision, bounded copy review/revision, loop promotion, and static page compilation. Current variants are scoring `v0.4`, candidate pool `guarded_packet_v2`, selector profile `slate_balanced_editor_v3`, selection review `selection_review_v1`, copy review `copy_review_v1`, and copy profiles `en_plain_v1`/`zh_matchnote_light_emotion_v1`. The candidate pool is split: `selectable_candidates` contains only public-card candidates, while `audit_candidates` contains progression, defensive, goalkeeper, hidden-gem, and match-coverage context that must not be selected directly. The selection loop must explicitly review whether selected cards are convincing, likely reader objections, alternative slates, weakest selected card, strongest omitted card, drop/replace verdicts, preferred card count, and the revision decision before promotion. The copy loop must review fact support, English flow, Chinese style, title core fact, display-name register, and unsupported claims before promotion. Compile fails unless `editorial_loop_summary.json` passes deterministic loop validation. The active slate is overall-first: pick the strongest 3-6 public cards when evidence supports them. Public award types are Player of the Day and Impact Pick only. Progression, defensive, goalkeeper, and hidden-gem metrics may appear in audit packets or as supporting evidence, but must not be standalone public labels. Do not add weak cards for match coverage; match coverage is a reader question, not a quota. A same match normally contributes at most two cards, but a dominant result with multiple top-ranked, independently strong candidates can justify a third. Public output is compiled `choices.json`, dated/index editorial HTML, and the homepage cards. Local audit output lives under `agent-runs/YYYY-MM-DD/` as deterministic packets, round-level selection/copy artifacts and validations, `final_selection_decision.json`, `final_copy.json`, canonical `selection_decision.json`, canonical `copy.json`, `editorial_loop_summary.json`, `editorial_loop_validation.json`, and `run.json`. The old cloud editorial queue and one-shot editorial review path have been retired; future API editor agents should be built against `docs/editorial-api-agent-spec.md` as shadow runs before becoming primary.

Default Editor's Choices scoring uses `config/scoring/v0.4.json`. The impact layer is derived from PMSR goals, lineup status, final scoreline, deterministic match-flow reconstruction, and supplemental FIFA timeline goal-involvement evidence for official assists. It can reward opening, equalizing, go-ahead, contextual match-winning, late, stoppage-time, late match-winning goals, comeback equalisers, comeback winners, only-goal winners, assists, goal involvements, braces, hat-tricks, and substitute scoring bursts. Progression, hidden-gem, defensive, and goalkeeper benchmark data may remain available for audit and supporting evidence, but not as standalone public award labels. Chinese player display names should come from `config/editorial/display_names/zh.json` when present; do not rely on ad hoc translation in public copy. Do not use media ratings or social reactions as direct scoring inputs.

Use `config/editorial/style_calibration/zh.jsonl` for recurring Chinese editorial taste feedback. Treat it as a curated bad/better example corpus for local Codex and future API editor agents, not as a broad banned-word list. Add high-confidence repeated issues there, keep one-off wording fixes in local copy, and only promote patterns to deterministic validation when the rule is stable.

The editorial workflow generates opinionated content, so prefer a PR branch over direct pushes to `main` unless the user explicitly asks for direct publication.

## Source Policy

- Do not commit `raw/*.pdf` or `raw/**/*.pdf`.
- Do keep source URLs, hashes, filenames, and timestamps in manifests and SQLite.
- Do keep FIFA timeline event provenance in `fifa_match_links`, `official_match_events`, and `goal_involvements` when assists or official event timelines are used.
- For any remote network operation, including `gh`, `git`, Vercel CLI, Supabase CLI, and similar tools, prefer the local machine proxy when one is configured; if `.env.local` already defines the proxy, load it and use it directly.
- Do not bypass login, CAPTCHA, rate limits, or access controls.
- If GitHub Actions fetching fails with 403/429, record the failure and use local assisted recovery instead of aggressive retrying.

## Recovery Workflow

When asked to check the project, start with:

```bash
python scripts/check_status.py
```

When a scheduled update fails:

1. Inspect recent GitHub Actions state with `gh run list --workflow update.yml --limit 5`.
2. Inspect the failing run with `gh run view <run-id> --log-failed`.
3. Download update artifacts if needed with `gh run download <run-id> --name update-status`.
4. Read `manifests/latest-run.json` and classify `failure_code`.
5. For `hub_fetch_failed`, `hub_parse_failed`, or dynamic page changes, inspect the FIFA hub with agent-browser and convert the finding into deterministic discovery code.
6. For `pdf_download_failed`, verify URL encoding, headers, and whether the source is temporarily unavailable.
7. For `pdf_parse_failed`, reproduce locally with the failed PDF, add or update a parser test, then patch the parser.
8. For `discovery_regression`, compare `manifests/discovered-sources.json` with the previous committed manifest before allowing data overwrite.
9. Run `python scripts/update_dataset.py`.
10. Run `python -m pytest`.
11. Commit the fix, push, and watch the replacement workflow run.

Agent-browser is a diagnostic fallback, not the normal CI path. The normal path should remain static HTTP discovery plus deterministic parsing.

## Testing

Use test-driven changes for parser and loader behavior. Parser changes should add a fixture-based or local-PDF test before implementation.
